# Kaleido MVP — Developer Brief
*Version 1.2 · March 2026*

**Scope of this doc:** System architecture — the two-system boundary, data contract, Lab pipeline, data models, and tech stack. This is the integration doc. For School UX, interaction rules, and student behaviour, read the UX Spec.

---

## 1. What Is Kaleido-MVP?

Kaleido-MVP is a test-prep SaaS for IELTS Writing Task 2. Its core insight: most test-prep apps hardcode content. Kaleido treats content as **dynamic data** — generated by an AI pipeline (the Lab), delivered by a web app (the School).

MVP target: Individual IELTS test-takers (B2C). Pilot with real users for qualitative feedback.

---

## 2. The Two-System Architecture

```
THE LAB  ──── prep-unit JSON (data contract) ────▶  THE SCHOOL
(AI pipeline)                                        (web app)
generates content                                    delivers content
```

**The Lab** — A Python pipeline that ingests IELTS questions, generates model essays, tags every sentence with rhetorical and structural metadata, and produces structured prep-unit JSON. The Lab determines *what* and *how much* to learn based on data analysis.

**The School** — A Remix web app that consumes prep-unit JSON and delivers a structured learning experience to students. The School is content-agnostic: it displays whatever the Lab generates.

**The glue** — The data contract (JSON schema) — a JSON schema document defining the exact structure the School needs the Lab to produce for one prep-unit. This is the prerequisite for both:
- School build (Remix routes cannot be specced without knowing the data shape)
- Lab build (pipeline cannot be built without knowing its output target)

The Lab and School are entirely decoupled through this contract. This is the most critical design boundary in the system.

**Responsibility split:** The School specifies the schema (what shape of data it needs). The Lab implements to that schema (selects content, generates practice items, structures output accordingly). The Lab owns content decisions within the structure the School defines.

---

## 3. MVP Scope

- **Test:** IELTS Writing Task 2 only
- **Users:** Individual test-takers (B2C)
- **Goal:** Students complete their assigned learning path and provide qualitative feedback on effectiveness
- **Not in scope yet:** B2B (centres), other IELTS tasks, multiple languages

## 4. Naming Convention (locked)

Practice names combine level + tool: e.g. `L4M` = Essay-level MCQ, `L1F` = Lexical Fill, `L2F` = Sentence Fill.

| Code | Meaning |
|------|---------|
| **L1** | Lexical — word/phrase level (lowest unit) |
| **L2** | Sentence |
| **L3** | Paragraph |
| **L4** | Essay — whole-essay level (highest unit) |
| **M** | MCQ (Multiple Choice) |
| **S** | Scramble |
| **F** | Fill Blank |
| **W** | Write |

**Note:** The prototype sample code (`school-mvp-prototype-sample_code.md`) has level codes inverted (L1=Essay, L4=Lexical). That is a prototype bug. The convention above is canonical.

## 5. The Data Contract

_Redesigned 2026-03-11. Replaces the previous nested JSON contract._

The Lab produces, and the School consumes, **prep-units** stored in a **7-table relational schema** (PostgreSQL via Supabase). The Lab's output is designed so the School can ingest it with a pure write — no transformation, no computation, no decisions. The ingestion step is: read JSON → INSERT rows → done.

### 5.1 Three-Layer Architecture

| Layer | Tables | Who writes | Who reads |
|-------|--------|-----------|-----------|
| **Content** ("Library") | `prep_unit`, `direction_ref` | Lab (P3) | School |
| **Logic** ("Playlists") | `tier_unit_sequence`, `qbank_unlocks` | Lab (P2/P3) | School |
| **Student** ("Tracking") | `student_path`, `student_unit_progress`, `student_attempt` | School | School + EC |

### 5.2 data.json Top-Level Shape (Pipeline 3 output)

```json
{
  "batch_id": "uuid — single content version identifier, generated per P2→P3 run",
  "prep_units": [ ... ],
  "tier_sequences": [ ... ],
  "qbank_unlocks": [ ... ],
  "direction_ref": [ ... ]
}
```

Each top-level array maps directly to one table. Ingestion = iterate + INSERT.

**`batch_id` semantics:** The School's `batch_id` is a **new UUID generated per P2→P3 pipeline run** — not P1's internal batch_id. It represents the complete curriculum package that gets locked to a student at onboarding. A new Lab run = new batch_id. Old students stay on old batch, new students get the new one.

### 5.3 PrepUnit — Content Storage

Each `prep_unit` row stores one IELTS essay/learning unit with two JSONB columns:

| Column | Type | What it holds |
|--------|------|---------------|
| `unit_id` | UUID PK | = `essay_id` from P1 |
| `batch_id` | UUID | School's content version |
| `question` | TEXT | IELTS question string |
| `structure_type` | TEXT | e.g. `"structure_4"` |
| `sentences` | JSONB | Essay display data — for peek modal, L4W reference panels, directions reading page |
| `practices` | JSONB | 12 pre-built practice objects (P0 through L4W) |

**Why JSONB for `practices`:** The School never queries individual practices — it always fetches the whole unit. JSONB is pragmatic: one query returns everything needed to render any phase. Breaks strict 1NF, but PostgreSQL handles it well and the query pattern justifies it.

**`sentences` JSONB shape** — per sentence:

```json
{
  "sentence_id": "uuid",
  "paragraph_type": "body_1",
  "order": 2,
  "canonical_text": "Full sentence text...",
  "rhetoric_tag": "explanation",
  "direction_tag": "sustainable_collective_forward",
  "lexical_items": [{ "phrase": "government funding", "pos": "NOUN" }],
  "syntax_items": ["Some argue that X, while others believe Y."]
}
```

No `similar_phrases` or `hint_sentences` in `sentences` — those are consumed at P3 assembly time and end up inside the `practices` JSONB (in L1F blank objects and L2F question objects respectively).

### 5.4 Practices — 12 Per Unit

The `practices` JSONB contains exactly **12 items** in fixed execution sequence:

`P0 → L4M → L3M → L2M → L1M → L1S → L2S → L3S → L4S → L1F → L2F → L4W`

Each item includes a `practice_code` field. Where retry exists, items include a `versions` array (V1, V2). P0, L2F, and L4W have no `versions` array (no retry).

| Tool | Required fields per version |
|------|-----------------------------|
| Cold Write (P0) | `{ "question": "IELTS question text", "prompt": "Write a complete..." }` — uses `question` from top level |
| MCQ (L4M, L3M, L2M, L1M) | `questions[]` each with `prompt`, `options[]` (text + correct flag) |
| Scramble — word bank (L1S, L2S) | `questions[]` each with `prompt`, `wordBank[]`, `blanks[]` (correct answers), `hint` |
| Scramble — drag-to-order (L3S, L4S) | `questions[]` each with `items[]` (id + text), `correctOrder[]` |
| Fill (L1F) | `sections[]` → `prompts[]` each with `text` (__ blanks), `hints[]` (POS tags), `similar_phrases[]`, `answers[]` |
| Fill (L2F) | `sentences[]` each with `text`, `hint_sentences[]`, `rhetoric_hint`. **No `answers[]`. No `versions`. No retry.** |
| Write (L4W) | `{}` — School renders using `question` from top level. No `versions`. |

> **`feedback` field removed** from all tool types. Wrong indicator only (red X / "incorrect"). The essay via peek is the only route to understanding.

> **`similar_phrases` in L1F:** Pre-computed by P2 via lexical embedding cosine similarity (0.5–0.7 range). Close enough to prompt recall, far enough to be wrong. School always shows them; no toggle.

> **L2F `hint_sentences`:** Pre-computed by P2 via sentence embedding cosine similarity (0.5–0.9 range, top-1 selection, cross-essay only). Not the correct answer. Used as recall cues only.

**MCQ distractor sourcing rule:** V1 and V2 distractors drawn from same tier pool at same **encoding level** (L1/L2/L3/L4). Lab pre-generates V1 and V2. School retry cycle: V1 → V2 → V1 (attempt 3 reuses V1).

**Fill grading:** Deterministic, no AI. Lab precomputes `answers[]`. School does case-insensitive string match. Applies to L1F only. L2F has no grading.

**Retry versions:** MCQ, Scramble (all), and L1F each have V1 and V2. P0, L2F, and L4W have no `versions` array.

## 6. The Lab — What Exists

The Lab is a working Python pipeline run in Google Colab, generating `data.json`.

**Location:** `kaleido-lab-mvp/`

**Three-pipeline structure (separate notebooks):**

- **Pipeline 1 — Generation** (slow, expensive): Calls an LLM per question. Generates model essays, tags sentences with rhetoric and direction metadata. Run once per question, or to regenerate.
- **Pipeline 2 — Power Law** (fast, pure math): Reads from DB and computes tier assignments (which units go to tier_50, which to tier_80) and Q Bank unlock mappings (which PrepUnit questions are unlocked by each tier unit, based on PoV direction coverage). All Q Bank questions are PrepUnits not assigned to the student's tier — they must have been through Pipeline 1 (to have essays with `direction_tag` data). Re-run whenever new questions are added.
- **Pipeline 3 — Assembly** (depends on P1 + P2): Needs tagged sentences from P1 and tier assignments + Q Bank mappings from P2. Assembles the final prep-unit JSON including all practice items and the `question_bank` array.

**External dependencies:** OpenAI API, Supabase Cloud

## 7. The School — Summary

The School is a **Remix web app** that delivers prep-units to students through a structured, sequential learning loop.

**For full School UX specification — including all interaction rules, practice flows, session model, timer rules, button states, retry logic, and Educator Console behaviour — read the UX Spec (`kaleido-school-ux-spec.md`). That document is the SSOT for all School behaviour.**

Summary of School structure per prep-unit:

```
P0 — Cold Write     Free essay on model essay question → Artifact 1 → EC immediately
P1 — Encoding       L4M → L3M → L2M → L1M → L1S → L2S → L3S → L4S → L1F → L2F
P2 — Applying       L4W — essay on model essay question → Artifact 2 → EC → unit gate
```

Tier selection at onboarding: student actively chooses **tier_50** (50 units) or **tier_80** (80 units). Content version locked to student at path assignment — never updated mid-path. Path ends when the last unit in the tier is completed; Question Bank remains open after path completion.

## 8. Data Storage

_Redesigned 2026-03-11. 7-table relational schema._

### 8.1 Content Layer — Lab writes, School reads

#### PrepUnit

One row per learning unit. See §5.3 for JSONB column shapes.

| Field | Type | Notes |
|-------|------|-------|
| `unit_id` | uuid PK | = `essay_id` from P1 |
| `batch_id` | uuid | School's content version (new UUID per P2→P3 run, not P1's internal batch_id) |
| `question` | text | IELTS question string |
| `structure_type` | text | e.g. `"structure_4"` |
| `sentences` | jsonb | Essay display data — peek modal, L4W panels, directions reading page. See §5.3. |
| `practices` | jsonb | 12 practice objects (P0 through L4W). See §5.4. |

**Who populates it:** Pipeline 3. Reads P1 Supabase tables + P2 enriched output, assembles JSONB columns, writes rows.

#### DirectionRef

Small reference table for the directions reading page. Maps internal direction IDs to human-readable argument text.

| Field | Type | Notes |
|-------|------|-------|
| `direction_id` | text PK | e.g. `sustainable_collective_forward` |
| `argument` | text | e.g. `"Environmental crisis demands government action"` |

**Who populates it:** Pipeline 2 or 3. Source: `DIRECTION_LOOKUP` dictionary (currently in P3, may move to P2).

---

### 8.2 Logic Layer — Lab writes, School reads

#### TierUnitSequence

Defines which prep-units belong to each tier, in what order, for a given content version. Not per-student — all tier_50 students on the same `batch_id` share the same sequence.

| Field | Type | Notes |
|-------|------|-------|
| `id` | uuid PK | |
| `batch_id` | uuid | School's content version |
| `tier` | text | `tier_50` / `tier_80` |
| `sequence_position` | int | 1-based — the order units are unlocked |
| `unit_id` | uuid FK → PrepUnit | |

**Constraints:**
- UNIQUE `(batch_id, tier, sequence_position)` — one unit per slot per content version
- UNIQUE `(batch_id, tier, unit_id)` — no duplicate units in same tier

**How the School uses it:** On onboarding, reads `TierUnitSequence` filtered by `batch_id + tier` to build the student's ordered unit list. Sequence is never copied per-student — resolved at query time via join.

**Who populates it:** Pipeline 2 (greedy algorithm). Each greedy step = one row.

#### QBankUnlocks

Maps each Q Bank question to the sequence position that unlocks it, plus the direction overlap that justifies the unlock.

**How Q Bank questions are derived:** Pipeline 2's greedy algorithm covers a target percentage of essays (50% for tier_50, 80% for tier_80). Within that pool, the greedy selects a subset as model essays (prep-units) — these are the questions the student fully encodes via all 12 practices. The remaining pool essays = Q Bank questions for that tier. Q Bank `question_id` = P1's `essay_id` — no new ID needed.

| Field | Type | Notes |
|-------|------|-------|
| `id` | uuid PK | |
| `batch_id` | uuid | School's content version |
| `tier` | text | `tier_50` / `tier_80` |
| `question_id` | uuid | = P1's `essay_id` for the Q Bank essay |
| `question_text` | text | Denormalised from P1's `essays.question` — avoids cross-DB join |
| `unlocked_by_sequence_position` | int | Earliest sequence step sharing a direction with this question |
| `shared_directions` | jsonb | Array of `direction_id` strings that overlap — lets Q-Bank UI show WHY a question relates to the unlocking unit. Join with `direction_ref` for display text. |

**Constraint:** UNIQUE `(batch_id, tier, question_id)`

**How the School uses it:** On unit complete, queries by `batch_id + tier + unlocked_by_sequence_position` to find newly available questions. Q Bank home uses `shared_directions` + `direction_ref` to display PoV context. Locked questions show "Unlock after completing Prep-Unit [N]".

**Who populates it:** Pipeline 2. Uses greedy output + direction overlap computation.

### 8.3 Student Layer — School reads + writes

#### StudentPath

Locked at onboarding, never mutated except `current_sequence_position`.

| Field | Type | Notes |
|-------|------|-------|
| `student_id` | uuid PK | |
| `teacher_code` | text | Links student to teacher in EC |
| `tier` | text | Permanent choice (`tier_50` / `tier_80`) |
| `batch_id` | uuid | Content version — locked forever at onboarding |
| `current_sequence_position` | int | Bookmark for "Resume Study" — starts at 1, increments on each unit complete, null when path complete |
| `created_at` | timestamp | |

> `exam_date` removed (locked 2026-03-08). `target_band` and `unit_count` removed — derivable from `tier`.

#### StudentUnitProgress

One record per student per unit. Tracks completion at phase boundaries only — the system never saves mid-practice state.

| Field | Type | Notes |
|-------|------|-------|
| `id` | uuid PK | |
| `student_id` | uuid FK → StudentPath | |
| `unit_id` | uuid FK → PrepUnit | |
| `status` | text | `locked` / `in_progress` / `complete` |
| `current_phase` | text | `p0` / `p1` / `p2` — which phase to resume at on return |
| `started_at` | timestamp | |
| `completed_at` | timestamp | Set when L4W is submitted (unit gate) |

**Constraint:** UNIQUE `(student_id, unit_id)`

#### StudentAttempt

Append-only. One record per item-attempt. Never updated.

> **Granularity — locked 2026-03-08:** One row per item-attempt (fine granularity). Pattern A (per-item immediate retry) means different questions within the same practice can be at different versions simultaneously. Fine granularity is the only shape that maps cleanly. EC signals (time, pass/fail, `failed_advanced`) are derived by aggregating item-level rows at query time.

| Field | Type | Notes |
|-------|------|-------|
| `id` | uuid PK | |
| `student_id` | uuid FK | |
| `unit_id` | uuid FK | Null for Q Bank writes |
| `qbank_question_id` | uuid | Null for unit practices. For Q Bank writes: the `question_id` from `qbank_unlocks`. |
| `practice_code` | text | `P0`, `L4M`…`L2F`, `L4W`, `QB_WRITE` |
| `item_id` | text | Specific question/item within the practice |
| `version` | text | `V1` / `V2`. Always `V1` for P0, L2F, L4W, QB_WRITE (no retry) |
| `response` | jsonb | Student's answer |
| `pass` | boolean | True = 100% correct. Null for P0, L2F, L4W, QB_WRITE (no pass/fail) |
| `failed_advanced` | boolean | True if advanced after 3 failures. Null for no-pass practices |
| `started_at` | timestamp | |
| `completed_at` | timestamp | |
| `artifact_type` | text | Null / `artifact_1` / `artifact_2` / `free_practice` |
| `artifact_content` | text | Essay text (write practices only — P0, L4W, QB_WRITE) |

> `peek_count` removed. EC does not receive peek data. `tool` and `level` columns removed — derivable from `practice_code`.

## 10. Key Design Principles

1. **Lab and School are fully decoupled** — the only interface between them is the prep-unit JSON schema.
2. **School is content-agnostic** — it renders whatever the Lab produces. No hardcoded content.
3. **Tools serve learning, not assessment** — the 4 practice types force active processing, not passive reading.
4. **Content reuse is a feature** — same essays appear across multiple prep-units at different cognitive entry points (spaced repetition by design).
5. **Versions = different angles, not harder** — retry logic gives new perspective, not increased difficulty.
6. **Content is the moat** — the interface is table stakes; the continuously-improving content engine is the differentiation.
7. **School specifies, Lab implements** — School owns the schema; Lab owns content decisions within it.
8. **No AI at grading time** — all grading is deterministic. Lab precomputes all answer sets. Fill grading is case-insensitive string match only. L2F and L4W have no grading.

## 11. Data Flow

_Updated 2026-03-11 to reflect relational schema redesign._

```
LAB (Python pipeline — 3 notebooks)
  P1: Generation (LLM, slow) → tagged sentences + rhetoric metadata → Supabase Lab DB
      OUTPUT: 6 tables (essays, sentences, essay_sentences, lexical_items, syntax_items, sentence_syntax)
      STATUS: ✅ Complete — no changes needed

  P2: Power Law + Similarity (math, fast) → reads P1 Supabase tables
      JOBS: greedy tier selection, similarity computation (similar_phrases + hint_sentences),
            Q Bank unlock mapping, direction overlap, DIRECTION_LOOKUP
      OUTPUT: tier_sequences[], qbank_unlocks[] (with shared_directions), direction_ref[],
              enriched snapshot (with similar_phrases + hint_sentences) for P3
      STATUS: ⚠️ Needs rebuild — same core logic, new output format

  P3: Assembly (depends on P1 + P2) → reads P1 tables + P2 output
      JOBS: assemble sentences JSONB, assemble 12-practice practices JSONB per unit
      OUTPUT: prep_units[] (with JSONB sentences + practices columns)
      STATUS: ⚠️ Needs rebuild

  ↓ produces
data.json (batch_id + prep_units + tier_sequences + qbank_unlocks + direction_ref)
  ↓ ingested by pure write — no logic, no transformation
School DB (PostgreSQL via Supabase, Prisma ORM) — 7 tables
  Content:  PrepUnit ← prep_units[], DirectionRef ← direction_ref[]
  Logic:    TierUnitSequence ← tier_sequences[], QBankUnlocks ← qbank_unlocks[]
  Student:  StudentPath, StudentUnitProgress, StudentAttempt (School-created)
  ↓ student signs up → chooses tier → batch_id + tier locked to StudentPath
School Remix App
  ↓ serves
Student UI (P0 → P1 Encoding → P2 Applying → Unit Complete → [loop or Path Complete])
  ↓ writes
StudentAttempt records (append-only — P0, P1 practices, L4W, QB writes)
  ↓ queried by
Educator Console (read-only — artifacts + signals + total accumulated time, filtered by teacher's students)
```
